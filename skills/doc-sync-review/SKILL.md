# Doc Sync Review

Doc Sync Review — удаление рассинхрона между реализацией и документацией. Это не генерация контента и не переписывание документации заново.

## Главный принцип

Актуализация документации — это drift removal. Документация после цикла должна отвечать: что теперь правда про систему.

Не улучшай текст без фактической причины. Не добавляй разделы "для полноты". Каждая изменённая строка должна иметь источник.

## Источники фактов

Используй источники в таком порядке:

1. Код, схемы, конфиги, `.env.example`, dependency files, тесты, runtime entrypoints.
2. Git diff/status, недавние коммиты, результаты валидации.
3. Принятый Human Plan или task plan как источник намерения и критериев приёмки.
4. Текущая документация как источник жанра и структуры, но не как источник истины.
5. Слова пользователя как источник факта, если они конкретны.

Если факт нельзя подтвердить, не записывай его в документацию. Сформулируй вопрос или дай draft с пометкой, что требуется подтверждение.

## Обязательный протокол

1. Прочитай локальные инструкции проекта (`AGENTS.md`, методологию docs, style guides).
2. Найди релевантный план/тикет/приёмку и текущий diff (`git status`, `git diff`, при необходимости `git log`).
3. Осмотри source of truth: entrypoints, API routes, configs, env vars, schemas, tests, dependency files.
4. Осмотри текущие docs. Не читай большие архивы целиком без причины; используй `rg` по терминам фичи и устаревшим словам.
5. Составь Doc Impact Map до правок.
6. Если локальные правила требуют явного разрешения на docs edits и пользователь его не дал, остановись на draft в чате.
7. Если правки разрешены, внеси минимальные изменения только в файлы из Doc Impact Map.
8. Проверь рассинхрон: повтори `rg` по устаревшим терминам, сверяй endpoints/env/config names/форматы ответа с кодом.
9. В финале укажи: что обновлено, что сознательно не тронуто, чем проверено, какие факты требуют подтверждения.

## Doc Impact Map

Перед редактированием зафиксируй короткую карту:

```text
File | Action | Why | Evidence
L1-architecture.md | update | transport changed SMB -> HTTP | main.py routes, plan validation
L2-functions.md | update | function contract changed | main.py diff
README.md | no change | no user-facing run/usage change | README already matches code
INSTALLATION.md | question | env variable unclear | code uses X, docs mention Y
```

`Action` должен быть одним из: `update`, `no change`, `draft only`, `question`.

## Жанры файлов

- `B0-context.md` — бизнес-цель, scope, ограничения. Меняй только при смене бизнес-смысла.
- `L1-architecture.md` — компоненты, интеграции, хранилища, внешние системы, потоки данных.
- `L2-functions.md` — функции/модули, входы, выходы, ветвления, ошибки, side effects.
- `README.md` — как понять, запустить и использовать проект.
- `INSTALLATION.md` — установка, зависимости, `.env`, deploy/setup.
- `ROADMAP.md` — статус milestone, следующие шаги, снятые/новые направления.
- `Plans/*` — намерение и история цикла. Не превращай план в отчёт; добавляй только минимальный статус/отклонение, если это явно нужно.
- `Archive/*` — исторический контекст. Не актуализируй как живую документацию без прямой просьбы.

Если файл имеет локальный жанр, следуй ему вместо этого списка.

## Правила правки

- Сохраняй структуру, заголовки и тон документа, если они не стали ложными.
- Заменяй устаревший факт новым фактом; удаляй устаревшее, если замены не нужно.
- Делай маленькие diff-правки. Не reflow whole file.
- Не добавляй объяснения реализации в L1 и бизнес-документы.
- Не документируй желаемое как реализованное.
- Не документируй отсутствие устаревшего поведения как текущий факт: не пиши "X больше не используется", "Y не поддерживается", если пользователь явно не просит migration/deprecation note. При drift removal просто замени старый факт на актуальный контракт или удали строку без замены.
- Разделяй `current`, `planned`, `historical`.
- В Markdown-документах используй Mermaid для схем, если схема действительно нужна и проект это предпочитает.
- В чате используй обычный текст или ASCII, если Mermaid может не отрендериться.
- Не редактируй документацию без разрешения, если project instructions это запрещают.

## Анти-паттерны

| Нет | Да |
|---|---|
| Переписать весь `L1` красивее | Заменить устаревшую схему транспорта |
| "Повышена надёжность" | "Запрос без PDF возвращает 400 до обращения к OCR" |
| Добавить общий раздел "Архитектурные преимущества" | Удалить SMB как актуальный транспорт |
| Описать будущий roadmap как текущий API | Пометить следующий тип документа как planned |
| Обновить `Plans/*` как живую архитектуру | Оставить план историей и обновить `L1/L2` |

## Проверка готовности

Doc-sync готов, когда:

- В живых docs нет устаревших утверждений о текущем режиме работы.
- Ключевые контракты совпадают с кодом: entrypoints, env vars, endpoints, input/output, storage, external systems.
- Каждая правка объясняется конкретным источником.
- Нет новых общих фраз без проверяемого содержания.
- Финальный ответ коротко перечисляет изменённые файлы и проверки.
