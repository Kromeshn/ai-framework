# SGR Debug Mode

Цель — корректный и воспроизводимый JSON-вывод через согласованную пару: системный промпт + JSON-схема.

## Scope

**Разрешено трогать:** `sgr/ocr_full.txt`, `sgr/ocr_amount_only.txt`, `sgr/schema_full.json`, `sgr/schema_amount_only.json`, параметры модели в `main.py` (`temperature`, `top_p`, `presence_penalty`, `max_tokens`).  
**Не трогать:** логику `main.py`.

---

## Механика: почему схема + промпт работают как тандем

На каждом шаге генерации происходит два действия одновременно. Промпт формирует **распределение вероятностей** по всему словарю токенов — это то, что модель *хочет* написать. Схема компилируется в грамматику и **обнуляет недопустимые токены** — это то, что модель *может* написать. Итоговый токен выбирается из пересечения: вероятности от промпта, обрезанные маской схемы.

**Следствие для проектирования:** промпт и схема закрывают разные типы ошибок и не заменяют друг друга. Ошибки смысла (что искать, как трактовать) — промпт. Ошибки формы (недопустимые значения, порядок шагов) — схема. **Правило конфликта:** если промпт предлагает значение, которое схема блокирует, модель молча выберет ближайший допустимый токен — результат валиден по форме, но семантически неверен. Конфликт предотвращается при проектировании: если поле допускает null — это должны говорить и схема, и промпт.

## vLLM + constrained decoding

| Что читает **движок** | Что читает **модель** |
|---|---|
| `type`, `enum`, `pattern`, `required`, порядок полей в `properties` | системный промпт + имена полей |
| `description` — **не доходит до модели** | — |

`description` — документация для человека, не канал управления. **Иерархия надёжности каналов** (от надёжного к ненадёжному): имена полей → имена полей + промпт → только промпт → только description. Имена полей в лидерах потому что в момент генерации значения имя поля стоит физически рядом с курсором — максимально локальная подсказка. Критичные правила для конкретного поля — в имя поля и поле-предшественник по Cascade. Глобальные политики — в системный промпт.

---

## Структура системного промпта

Промпт должен содержать три блока. **Блок 1 — контекст:** кто ты, что за документ, ожидаемое качество OCR. **Блок 2 — политика при неопределённости (критичный):** required-поля создают давление — без явной инструкции модель галлюцинирует вместо честного null. Прописывать явно: «если реквизит не найден — null, не додумывай», «если OCR нечитаем — сигнализируй через confidence_low». **Блок 3 — доменные правила:** то, что схема не умеет кодировать — ИНН продавца всегда в шапке, дата документа ≠ дата отгрузки, «Всего к оплате» ≠ «Сумма без НДС».

## SGR-паттерны

| Паттерн | Суть | Триггер |
|---|---|---|
| **Cascade** | Порядок полей = порядок мышления. Каждое написанное поле становится контекстом для следующего — commitment effect. | Всегда, базовый паттерн |
| **Routing** | Поле-дискриминатор первым задаёт контекст для всех последующих полей. | Разные кейсы требуют разной логики |
| **Cycle** | Массив с `minItems`/`maxItems` для неизвестного количества элементов; внутри каждого — Cascade. | Количество элементов неизвестно заранее |

Паттерны комбинируются: Routing определяет тип документа → Cascade обрабатывает каждый реквизит → Cycle собирает строки таблицы → внутри каждой строки снова Cascade.

---

## Техники реализации Cascade

Инструменты организации промежуточных полей внутри Cascade. Промежуточное поле всегда стоит **до** финального — иначе превращается в постфактум-обоснование, а не рассуждение.

| Техника | Суть | Триггер |
|---|---|---|
| **Evidence-First** | Первое поле — дословная цитата из источника, до интерпретации. Цепочка линейная: `_raw` → нормализованное. | OCR-шум, неоднозначный формат, риск подмены значения |
| **Self-Check до** | Поле-булеан «условие выполнено?» перед финальным. Барьер перед ошибкой. | Риск галлюцинации, поле может отсутствовать в документе |
| **Self-Check после** | Верификационное поле с enum после уже выданного значения. Не исправляет — детектирует и маркирует для ручной проверки. | У поля есть структурно похожие конкуренты в документе; нестабильные метрики на выборке |
| **Decompose-and-Aggregate** | Одна сырая строка расходится на несколько независимых полей-частей. Ограничения на частях (pattern) делают невозможным перепутать их местами. | Одна строка источника должна разбиться на ≥2 структурно разных поля |

**Отличие Evidence-First от D&A:** Evidence-First — цепочка линейная (одна строка → одно нормализованное значение). D&A — цепочка расходится (одна строка → несколько независимых полей).

Поля бывают двух видов: **поля-данные** (нужны в финальном ответе) и **поля-черновики** (нужны только для мышления модели, в пайплайн не попадают). В dev-режиме оставляй все поля — по промежуточным видно где модель свернула не туда. В prod фильтруй через allowlist: `result = {k: v for k, v in raw.items() if k in FINAL_FIELDS}`.

## Правила проектирования схемы

**Имена полей — основная инструкция модели внутри схемы.** `total_amount_row_found` говорит больше чем `flag1` — имя поля физически стоит перед курсором в момент генерации значения.

**`minItems: 1` по умолчанию, `minItems: 2` только если 100% уверен** что минимум два элемента всегда есть — иначе модель изобретает элементы чтобы удовлетворить ограничение.

**enum только для конечных множеств** (типы документов, статусы, алфавит). Никогда на имена организаций, адреса, свободный текст — модель выберет «ближайший» вариант вместо честного значения.

**Плоская структура лучше глубокой вложенности.** `seller_address_city` надёжнее чем `seller.address.city` — глубина >3 уровней снижает качество и увеличивает время компиляции грамматики.

**`["string", "null"]` + `required` лучше чем опциональное поле.** Семантически то же самое, но для компилятора грамматики принципиально проще.

**line_number как якорь в Cycle.** В массиве items каждый элемент должен начинаться с `line_number` — фиксирует о какой строке модель говорит. При плохом OCR сразу видно в логах если строка пропущена или переставлена.

## Антипаттерны

| Антипаттерн | Симптом | Решение |
|---|---|---|
| enum на высоковариативном поле | Модель пишет «ближайший» вариант вместо реального значения | enum только для конечных множеств |
| `minItems` > реального минимума | Модель дублирует или изобретает элементы массива | `minItems: 1` по умолчанию |
| Вложенность >3 уровней | Медленная компиляция, снижение качества | Плоские поля с семантическими именами |
| Конфликт промпт/схема | Валидный JSON, но семантически неверное значение | Привести промпт и схему к одной политике |
| required-поле без политики null в промпте | Галлюцинация вместо честного null | Явно прописать политику неопределённости в промпте |

## Рабочий протокол

1. Прочитать `docs/B0-context.md` — бизнес-контекст документов
2. Прочитать текущий промпт и схему
3. Найти проблемное поле → определить источник ошибки (промпт или схема)
4. Выбрать SGR-паттерн(ы) и технику Cascade по триггеру
5. Согласовать изменения: что уходит из промпта → что переезжает в схему, нет ли конфликта
6. Внести изменения в промпт и схему согласованно
7. Тестировать на проблемных кейсах из датасета
8. При стабильном результате — коммит

```bash

# Примеры:
# sgr: SGR-каскад total_amount_row_found → total_amount_raw → total_amount
# sgr: self-check-после total_amount_check для ручной проверки

git log --oneline sgr/      # история итераций
git checkout <hash> -- sgr/ # откат при регрессии
```
