# Implementation Review

Ревью реализации после Agent Plan/Human Plan. Это не ревью плана, а проверка фактического кода и поведения.

Не исправляй код без явной просьбы пользователя. Начинай с findings, не с пересказа изменений.

## Что является источником истины

1. Human Plan — источник истины по поведению, scope, границам и критериям приёмки.
2. Agent Plan — технический чеклист: какие файлы, функции, тесты и шаги агент обещал сделать.
3. Код и diff — источник истины по фактической реализации.
4. Тесты и ручные проверки — подтверждение, что поведение наблюдаемо.
5. Локальные правила проекта и `dignified-python` — источник требований к качеству Python.

Если Human Plan и Agent Plan расходятся, приоритет у Human Plan. Agent Plan не может легализовать неправильное поведение.

## Workflow

1. Найди релевантный Human Plan. Если он не указан и не обнаруживается уверенно, спроси один конкретный вопрос.
2. Найди Agent Plan, если он есть в чате, плане задачи или передан пользователем. Если его нет, ревьюй против Human Plan и diff, явно скажи об этом.
3. Проверь `git status` и diff. Не откатывай чужие изменения.
4. Прочитай только файлы, нужные для понимания изменённого поведения.
5. Запусти доступные фокусные проверки: tests, import/compile, lint/type check. Не устанавливай инструменты без просьбы.
6. Сверь реализацию по слоям:
   - поведение и scope против Human Plan;
   - существенные технические шаги против Agent Plan;
   - runtime-баги, edge cases, регрессии, API-контракты, swallowed exceptions;
   - покрытие тестами и соответствие критериям валидации;
   - качество Python: явные проверки, локальные изменения, type hints, отсутствие лишних абстракций, `dignified-python`.
7. Сообщи findings с файлом и строкой. Если проблем нет, скажи это прямо и назови остаточный риск.

## Приоритеты findings

- `P0` — реализация не запускается, ломает основной сценарий или нарушает критичный контракт.
- `P1` — поведение противоречит Human Plan, есть серьёзный runtime-риск или регрессия.
- `P2` — пропущен важный edge case, тест, часть Agent Plan или критерий приёмки.
- `P3` — локальная maintainability/quality проблема, которая может стать ошибкой позже.

Не репорть вкусовые рефакторы, широкие переписывания и замечания вне scope.

## Формат ответа

Читатель ревью — владелец проекта, не инженер. Пиши так, чтобы по findings
можно было принять решение без встречных вопросов «а это опасно?» и «что делать?».

Начинай с итога одной строкой: есть ли что чинить обязательно — и что именно.

Каждый finding — абзац из трёх ответов по порядку:

- **что случилось** — поведением системы, бытовым языком;
- **чем грозит** — последствие и частота словами («файл уедет циклом позже; редкость»);
- **что делать** — вердикт: 🔴 чинить обязательно · 🟡 чинить потом (когда) · ⚪ не чинить, знать.

Файл и строка — ссылкой, severity (P0-P3) — в скобках после вердикта.
Термин — с расшифровкой в той же фразе; механика — после поведения, не вместо.

Для findings P2 и критичнее подключи скилл `explanation-crew` (через Skill tool)
и разверни каждый такой finding в его регистре: медленно, с нуля, с конкретным
примером «как это проявится». P3 хватает короткого абзаца.

Затем коротко: `Test Gaps`, `Checks Run`, `Residual Risk`.
Нет findings — скажи прямо, назови проверки и непроверенные зоны.
