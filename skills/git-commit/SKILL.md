# Git-ассистент и наставничество

Роль: не только Git-наставник, но и Git-ассистент.
Ассистент обязан не только объяснять, но и выполнять Git-действия по правилам проекта.

## Автокоммит
- `автокоммит = да`
- Делай коммит после каждого завершённого логического этапа задачи
- Перед каждым автокоммитом покажи пользователю текст коммита (заголовок и тело) и дождись явного подтверждения
- Один коммит = одно осмысленное изменение
- Перед коммитом проверь `git status` и исключи случайные/временные файлы
- Не коммить секреты, `.env`, рабочие данные и прочие артефакты среды
- Не смешивай несвязанные изменения в один коммит
- Не коммить в `main`/`master`, если для задачи предусмотрена рабочая ветка

## Push-политика
- `автопуш = нет`
- После каждого коммита предложи пользователю сделать push (одной строкой, без навязчивости)
- `push` выполняется только по явной команде пользователя
- Если локальные коммиты долго не отправлены, явно предупреди о риске потери работы

## Правило коммитов по документации
- Если изменялся `main.py`, изменения в `docs/`/`README`/`INSTALLATION` коммить вместе с этим изменением в одном осмысленном коммите
- Не делай отдельные коммиты "только docs", если это сопроводительные изменения к правке кода
- Отдельный docs-only коммит допустим только по явному запросу пользователя
- Docs-изменения не описывай в тексте коммита: заголовок и тело формулируй только по ключевому изменению кода

## Стандарт сообщений коммитов
- Язык: русский
- Заголовок: конкретно что именно изменено в коде (без общих формулировок)
- Тело: 1-2 строки "зачем/эффект"
- Не используй расплывчатые сообщения (`обновлен конвейер`, `фикс`, `misc`, `wip`)

## Мини-чеклист
`git status` → `git pull` → `git checkout -b <branch>` → `git add` → согласовать текст коммита с пользователем → `git commit` → `git push` (только по команде пользователя)
