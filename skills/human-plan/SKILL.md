# Human Plan

Human Plan — аналитический мост между бизнес-интентом и реализацией. Фиксирует целевое поведение фичи до Agent Plan и кода.

`Business Intent → Sensemaking → Human Plan → Agent Plan`

Human Plan отвечает: что будет работать, зачем, где границы, как увидеть результат. Agent Plan — как это реализовать.

## План пишут для читателей, а не «для записи»

У готового плана три читателя, и читают они в разное время на разную глубину:

- **Автор через недели** — перечитывает, чтобы вернуть в голову модель фичи: что нарешал и зачем. За минуты, по верхушке.
- **Агент в новой сессии** — дискавери не видел; читает план целиком один раз, ему нужна нормативка для реализации.
- **Холодный чтец будущего** — контекст потерял или не имел; хочет понять решения и доказательства под ними.

Главная ловушка: план, отсортированный по **порядку, в котором рождались мысли** (контекст → логика → детали → решения где-то внизу), хоронит то, что читателю нужно первым. Автор через месяц ищет «что я нарешал», а натыкается на отчёт о разведке и тонет. Поэтому план сортируется **по читателю и частоте чтения**, а не по типу контента.

## Две зоны

**Верх — реактиватор.** Возвращает ментальную модель за один проход: что и зачем; схема потока; ключевые решения; что осталось открытым. Это всё, что нужно автору, чтобы сказать «вспомнил, согласен» или «вот тут ещё подумаю».

**Низ — детали.** Разворачивание для агента: правила поведения, состав артефактов, конфиг, границы, валидация, доказательства. Сюда заходят, когда верха не хватило.

**Инвариант: низ только разворачивает верх.** Ниже нет ни одного решения, которого нет наверху — только его раскрытие. Тогда реактивация по верхушке честна: автор по построению ничего не пропускает.

Не два файла. Два разойдутся при первой правке, и возникнет дыра: автор утвердил выжимку, агент исполняет полный — тождество никто не гарантирует. Один файл, две зоны, граница — горизонтальная черта (без ярлыка-метафоры вроде «подвал»; зона опознаётся по содержанию, не по вывеске).

## Решения — таблицей, с колонкой «Почему»

Ключевые решения — одна таблица `Решение | Почему`. **Одна, не две** (верхняя «короткая» + нижняя «полная» разъезжаются и дают дубль-дрейф — то самое, чего избегаем отказом от двух файлов).

«Почему» — не полное обоснование, а **причина-крючок**: то, за что решение защёлкивается обратно в голове. Через недели голое «делаем X» выглядит произвольным, мозг буксует и продрачивает заново. Крючок снимает буксовку: «а, точно, поэтому».

### Как писать «Почему»

- **Каждая фраза несёт новое.** Delete-тест: вычеркни фразу — что потерялось? Ничего → вон. «Это нормальный исход, а не сбой» вычёркивается без потери, если в названии решения уже сказано «→ 200, не ошибка».
- **Закрывает встречный вопрос, а не приглашает его.** «3k символов — заголовок в первой строке» приглашает «а если не в первой?». Полный якорь закрывает: «3k — заголовок вверху (9/9 на выборке); склейку ширина окна всё равно не лечит — отдельный механизм».
- **Плотно, но не морзянка.** Полные слова, без закодированных сокращений (`×7`, «окно», «узел» без расшифровки).
- **Цифры вместо прилагательных.** Не «разумный потолок» / «дорого», а «9 Мпикс ≈ А4 в 300 точек/дюйм», «страница ≈ 8 600 токенов из 60 000».

До → после (одна строка таблицы):
- ✗ `Текст целиком — выборка влезает в окно ×7, локализация = лишний узел`
- ✓ `Текст целиком — самый длинный договор 20 тыс. знаков, модель читает за раз ~150 тыс.; резать нечего`

## Форма следует задаче раздела

Где контекст позволяет — **визуализируй**: таблица или схема читается лучше абзаца. Подбирай форму по работе раздела:

- поток / маршрут → **диаграмма** (Mermaid в файле);
- выбор и его причина → **таблица решений**;
- «в ситуации S поведение B» → **таблица правил**;
- значения, ключи, дефолты → **таблица конфига**;
- текст абзацем — только для того, что не ложится ни в одну структуру.

**Решение живёт в таблице решений, не в тексте абзацем.** Если читатель ищет «что нарешали» — он смотрит в таблицу решений, а не вычитывает прозу. Проза (описание механики) разворачивает решения, но не прячет их в себе.

Мелочи читаемости: **жирным — только слово-зацепка** в правилах/рисках (по которому пункт опознаётся), чтобы план сканировался по акцентам; короткие предложения без канцелярита («является», «осуществляется», «в связи с необходимостью»).

## Какие разделы обязательны

Обязательны только **смысловые** разделы — те, без которых план не план: что/зачем, схема (если есть поток), решения, как увидеть результат (валидация), границы. Доп. разделы модель добавляет **по контексту фичи** — рамка их не запрещает и не предписывает (например, у SGR-фичи появляются «SGR-состав» и «Конфигурация», у фичи без них — нет).

Типичный скелет (пример, не шаблон):

```
ВЕРХ:  Шапка (Что/Зачем/Не делаем) · Схема · Решения · [Открытые вопросы]
———
НИЗ:   Контракт/правила поведения · артефакты · конфиг · границы (OUT+триггеры) · валидация · доп. контекст
```

## Чего в плане быть не должно

- **Комментариев о самих разделах.** «Этот раздел опционален», «здесь разворачивание верха» — это метаданные *про форму*, их место в этом скилле, а не в каждом экземпляре плана. В плане раздел либо есть с содержанием, либо его нет.
- **Раздела «Открытые вопросы» как затычки.** Туда идёт только то, что *явно просили записать* или что *сознательно отложили с условием возврата*. Если что-то вывалилось и не закрыто — закрывают в обсуждении, а не паркуют фейковым вопросом. Нет реальных вопросов → нет раздела.
- **Доказательств как обязаловки.** Разведка, замеры, эксперименты — опциональный нижний слой; есть, когда их проводили. Не каждый цикл их имеет.

## Форму диктует содержание, не счётчик

Размер раздела определяет цикл, не лимит. Короткий цикл — решения в три строки. Большой скоуп — таблица решений на 15 строк, и это нормально; не дробить и не прятать ради «покороче». Пустой раздел-плацебо хуже отсутствующего. Не ограничивай форму числом строк — упор на смысл.

## Поведение при написании

- **Факты из проекта сперва, потом вопросы.** Прочитай код, доки, git — спрашивай только то, чего там нет. Не выдавай проекцию за факт.
- **По одному существенному вопросу.** Не вали список.
- **Дави на туман.** «Корректно», «гибко», «при необходимости», «обработать ошибки» — дыры. Заменяй наблюдаемым поведением: не «корректно обрабатывает», а «если реквизит отсутствует — поле становится null».
- **Маркируй факт vs гипотезу.** «Подтверждено на выборке» и «предполагаем» несут разный вес; гипотеза не должна выглядеть фактом.

## Уровень

Поведение системы, не реализация. Компоненты, потоки, правила, состояния, последствия, валидация. Без кода и псевдокода. Не называй файлы, функции, классы, библиотеки — кроме случаев, когда это продуктовый компонент или граница системы. Механику (как именно: `ALTER` vs пересоздание, какой класс) уноси в Agent Plan — в Human Plan живёт ограничение («прод-журнал должен пережить деплой»), не способ.

## Анти-паттерны

| Нет | Да |
|---|---|
| «Используем pandas DataFrame для индексации» | «Журнал заказов матчится по трём ключам» |
| «Корректно обрабатывает ошибки» | «Если реквизит отсутствует, поле становится null» |
| «Гибкий порог» | «Порог 5 страниц, меняется через config» |
| «Почему» с закодированным сокращением (`×7`, «узел») | «Почему» обычными словами и цифрами |
| Решение спрятано в абзаце прозы | Решение — строкой в таблице решений |
| Две таблицы решений (короткая + полная) | Одна таблица, каждое решение один раз |
| Параллельные списки (in/out, до/после) | Двухколоночная таблица — сравнивается одним взглядом |
| Раздел «Открытые вопросы» ради заполнения | Раздела нет, если реальных вопросов нет |
| Комментарий «этот раздел опционален» внутри плана | Содержание есть или раздела нет |
| Разведка/замеры в начале файла | Доказательства — опциональный низ |

## Готовность

План готов к Agent Plan, когда:

- по верхушке понятно, что делает фича на конкретном входе и почему так — без похода вниз, чтобы вспомнить;
- низ разворачивает верх и ничего нового не вводит (инвариант держится);
- границы scope явные;
- открытые вопросы закрыты или сознательно отложены с условием возврата;
- валидация наблюдаемая, не «проверить, что всё работает».

Перед Agent Plan проверь план на дыры, противоречия, неясный scope. Не пиши Agent Plan без просьбы.

## Главное правило

Human Plan не доказывает, что агент сможет кодить. Он собирает устойчивую модель будущего поведения фичи — такую, что её можно вернуть в голову за минуты и развернуть в детали по требованию.
