# Human Plan

Human Plan — аналитический мост между бизнес-интентом и реализацией. Фиксирует будущее поведение фичи до Agent Plan и кода.

## Цепочка

```mermaid
flowchart LR
    BusinessIntent["Business Intent"]
    Sensemaking["Sensemaking"]
    HumanPlan["Human Plan"]
    AgentPlan["Agent Plan"]

    BusinessIntent --> Sensemaking --> HumanPlan --> AgentPlan
```

Human Plan отвечает: что будет работать, зачем, где границы, как увидеть результат.
Agent Plan отвечает: как агент будет это реализовывать.

## Поведение модели

- Сначала ищи факты в проекте, потом спрашивай пользователя.
- Спрашивай по одному существенному вопросу.
- Дави на туман: "корректно", "гибко", "при необходимости", "обработать ошибки".
- Держи уровень поведения системы, не уровень реализации.
- Решение фиксируй сразу: решение + одна строка "почему".
- Сложный план собирай секциями; простой можно дать одним сухим draft.
- После драфта взгляни на план чужими глазами. Если суть не схватывается за несколько секунд, глаз скользит по однотипной стене буллетов, ключевые слова не выпрыгивают — примени техники из секции «Подача». Если плану на 30 строк не нужен ни TL;DR, ни таблицы — не лепи их ради единообразия.

## Уровень

- Компоненты, потоки, правила, состояния, последствия, валидация.
- Без кода и псевдокода по умолчанию.
- Не называй файлы, функции, классы, библиотеки, если это не продуктовый компонент или системная граница.
- Технические детали уходят в Agent Plan.

## Обязательное ядро

- **Контекст** — что сейчас не так, зачем меняем, какой эффект нужен.
- **Целевая логика** — как фича должна работать с точки зрения системы/пользователя.
- **Границы** — in scope и явный out of scope.
- **Валидация** — как пользователь или агент увидит, что фича работает.

## Опционально

Добавляй только по смыслу:

- **Поток** — Mermaid-схема или короткие шаги.
- **Правила** — нетривиальные развилки, приоритеты, бизнес-условия.
- **Решения** — принятые архитектурные решения, по одной строке "почему".
- **Риски и края** — существенные пустые состояния, ошибки, неоднозначности.
- **Открытые вопросы** — блокирующие или сознательно отложенные, с условием возврата.

Пустая секция-плацебо хуже отсутствующей секции.

## Форма

- Полевая шпаргалка, не статья.
- Максимальная плотность: если предложение не меняет смысл, удалить.
- Mermaid — предпочтительный формат схем для документации и Markdown-артефактов.
- ASCII — только в чате или там, где Mermaid не рендерится.
- Если в проекте есть doc-конвенция, следуй ей; иначе имя файла короткое, говорящее, kebab-case.
- Валидацию оформляй чеклистом `- [ ]`; предварительные условия для проверки ставь первыми пунктами валидации.

## Подача

Техники для читаемости длинных планов. Применяй когда тянут вес для конкретного плана — не как обязательный набор.

- **TL;DR-блок сверху** (3 строки в blockquote: что / зачем / чего не делаем). Имеет смысл, когда план разворачивается на несколько экранов и читателю нужно за несколько секунд схватить суть, прежде чем нырять в детали. Короткому плану — лишний слой.
- **Параллельные списки → таблица.** Если у двух списков симметричная структура (in/out scope, решение/почему, до/после), двухколоночная таблица позволяет сравнить их одним взглядом. Списки разной природы оставляй буллетами.
- **Bold на ключевых словах** в Правилах и Рисках. Жирный — точка зацепления для быстрого сканирования: прочитав только акценты, читатель пересобирает смысл. Не жирнить ради жира — только то слово, по которому правило идентифицируется.
- **Короткие предложения без канцелярита.** Конструкции вроде «На основе текущего поведения», «является», «осуществляется», «в связи с необходимостью» удлиняют без смысла. Заменяй простыми формулировками.

## Готовность

Human Plan готов к Agent Plan, когда:

- по плану понятно, что делает фича на конкретном входе/сценарии;
- границы scope явные;
- открытые вопросы закрыты или сознательно отложены;
- валидация наблюдаемая, не "проверить, что всё работает".

Не пиши Agent Plan без просьбы пользователя.
Перед Agent Plan сначала проверь Human Plan на дыры, противоречия и неясный scope.

## Анти-паттерны

| Нет | Да | Почему |
|---|---|---|
| "Используем pandas DataFrame для индексации" | "Журнал заказов матчится по трём ключам" | Уровень реализации вместо поведения |
| "Корректно обрабатывает ошибки" | "Если seller_inn отсутствует, division_name становится null" | Туман vs наблюдаемое поведение |
| "Гибкий порог" | "Порог — 5 страниц, меняется через config" | Конкретика вместо абстракции |
| "Обновить parse_document()" | "OCR-результат нормализуется перед обогащением" | Поведение, не имя функции |
| Пустой раздел "Риски" | Удалить раздел | Плацебо размывает фокус |
| In scope и Out of scope двумя списками подряд | Двухколоночная таблица `In \| Out` | Симметричная структура сканируется одним взглядом |
| Список из 7 решений с «*Почему:*» в конце каждой строки | Таблица `Решение \| Почему` | Параллельная структура даёт быстрый скан первого столбца |
| Стена буллетов одной плотности | Короткие предложения с **bold** на ключевых словах | Глаз цепляется за акценты, F-pattern reading работает |

## Главное правило

Human Plan не доказывает, что агент сможет кодить. Он собирает устойчивую модель будущего поведения фичи.
